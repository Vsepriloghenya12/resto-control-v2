# Live-журнал Resto Control v2

## Запись 001 — Новый принцип работы

Приложение Resto Control v2 пишется с нуля. Старый проект используется как источник бизнес-логики, сценариев и визуального характера, но не как база для хаотичных правок.

Работа идёт по страницам:

1. смысл и функционал;
2. визуал;
3. архитектура данных и API;
4. код;
5. проверка сценария.

Визуал должен быть аккуратным, рабочим, ресторанным, профессиональным. Можно использовать подходы shadcn/v0, но без лишней декоративности и случайных эффектов.

## Запись 002 — Первая страница

Первая страница — вход и создание ресторана.

Сценарии:

- войти в существующий ресторан;
- создать новый ресторан.

Сотрудник не создаёт ресторан сам. Он входит по телефону/email и паролю после приглашения или создания учётной записи владельцем/менеджером.

Архитектура:

- `User` — человек;
- `Restaurant` — заведение;
- `Membership` — связь пользователя с заведением и ролью;
- `Session` — текущий вход.

Роль не хранится просто в `User`. Роль зависит от ресторана через `Membership`.

## Запись 003 — Утверждённый визуальный референс

Desktop:

- split-layout;
- слева тёмный бренд-блок с ресторанным фоном;
- справа светлая карточка формы;
- вкладки «Войти» / «Создать ресторан»;
- оранжевая главная кнопка;
- убрать блок «Нет аккаунта?»;
- убрать вход по QR-коду.

Mobile:

- отдельный компактный рабочий экран входа сотрудника;
- логотип сверху;
- крупный заголовок Resto Control;
- подзаголовок «Рабочий вход для сотрудника»;
- карточка формы;
- два поля;
- ссылка «Забыли пароль?»;
- кнопка «Войти»;
- нижняя подпись «Нет доступа? Обратитесь к управляющему.».

Реализация должна быть максимально близкой к утверждённому референсу.
